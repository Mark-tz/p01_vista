## 2. 可用性需求

### 2.1 可用性目标

| 服务等级 | 可用性 | 年停机时间 | 适用服务 |
|----------|--------|-----------|----------|
| Tier-1 | 99.9% | < 8.76h | 实时监控、远程操控、设备通信 |
| Tier-2 | 99.5% | < 43.8h | 任务调度、告警中心 |
| Tier-3 | 99.0% | < 87.6h | 数据中心、报表、OTA |

### 2.2 高可用设计

| 组件 | HA 方案 | RPO | RTO |
|------|---------|-----|-----|
| API Gateway | 多副本 + LB | 0 | < 10s (Pod 重调度) |
| 业务服务 | 多副本 + 无状态 | 0 | < 30s |
| PostgreSQL | 主从复制 + 自动故障转移 | < 1min | < 30s |
| Redis | Sentinel / Cluster | < 1s | < 10s |
| EMQX | 集群 (≥ 3 节点) | 0 | < 5s |
| Kafka | 多副本 (RF=3, ISR=2) | 0 | < 10s |
| MinIO | 纠删码 (EC) | 0 | < 10s |
| InfluxDB | 集群 / 备份 | < 1h | < 1h |

### 2.3 故障处理策略

| 故障场景 | 影响 | 处理策略 |
|----------|------|----------|
| 单服务 Pod 崩溃 | 该服务短暂不可用 | K3s 自动重启 + 多副本接管 |
| 数据库主节点故障 | 写入暂停 | 自动故障转移至从节点 |
| Redis 节点故障 | 缓存失效 | Sentinel 自动切换 + 缓存穿透保护 |
| EMQX 节点故障 | 部分设备断连 | 客户端自动重连至其他节点 |
| 网络分区（设备侧） | 设备指令不可达 | MQTT QoS + 离线队列 + 设备本地自治 |
| 全站断电 | 系统不可用 | UPS 保护 → 优雅关闭 → 来电自动恢复 |
| 云端不可达（混合部署） | 边缘自治 | 本地缓存 + 断网续运 + 恢复后数据同步 |

### 2.4 降级策略

| 场景 | 降级方案 |
|------|----------|
| AI 服务不可用 | 跳过 AI 识别，仅存储原始图片，后续补识别 |
| 视频流异常 | WebRTC → HLS → 仅抓拍 逐级降级 |
| 数据库读压力大 | 降级到 Redis 缓存数据（可能非最新） |
| 消息队列堆积 | 丢弃低优先级遥测，保留告警和任务消息 |

## 3. 可扩展性需求

### 3.1 扩展维度

| 维度 | 当前目标 | 3 年目标 | 扩展方式 |
|------|----------|----------|----------|
| 站点数 | 1-5 | 50+ | 多集群 / 混合部署 |
| 单站机器人 | 1-5 | 20+ | EMQX 集群扩展 |
| 并发用户 | 50 | 500+ | 服务水平扩展 |
| 数据存储 | 1 TB | 50 TB+ | 存储集群扩展 |
| AI 模型 | 3-5 | 20+ | GPU 节点池扩展 |

### 3.2 扩展性设计原则

- **服务无状态**：业务服务不保存本地状态，支持水平扩缩容
- **数据分片**：大表按 site_id 分区，InfluxDB 按时间分片
- **异步解耦**：核心链路通过 Kafka 异步解耦，避免级联故障
- **缓存分层**：L1（本地缓存 10s）→ L2（Redis 5min）→ L3（数据库）
- **弹性伸缩**：基于 CPU/内存/QPS 的 HPA 策略

## 4. 安全需求

### 4.1 安全分层

| 层级 | 安全措施 |
|------|----------|
| **网络层** | 网络分区隔离、防火墙规则、DDoS 防护、VPN/专线 |
| **传输层** | 外部 TLS 1.3、内部 mTLS、MQTT TLS |
| **认证层** | JWT + Refresh Token、设备 X.509 证书、MFA（可选） |
| **授权层** | RBAC 权限模型、资源级访问控制、API 级鉴权 |
| **应用层** | 输入校验、SQL 注入防护、XSS 防护、CSRF Token |
| **数据层** | 敏感字段加密 (AES-256)、数据库 TDE、备份加密 |
| **审计层** | 全操作审计日志、登录日志、异常行为检测 |

### 4.2 合规要求

| 要求 | 措施 |
|------|------|
| 等保 2.0 三级 | 按等保要求落实安全基线 |
| 数据分级 | 定义数据敏感等级（公开/内部/机密/绝密） |
| 日志保留 | 审计日志保留 ≥ 180 天 |
| 密码策略 | 最低 8 位、含大小写+数字+特殊字符、90 天过期 |
| 会话安全 | 30 分钟无操作自动登出、并发登录限制 |
| 接口安全 | API 限流、请求签名、IP 白名单（可选） |

### 4.3 设备安全

| 措施 | 说明 |
|------|------|
| 设备认证 | 设备证书 + 唯一 ID 双因素 |
| 指令签名 | 关键指令附加 HMAC 签名 |
| 固件签名 | OTA 包 RSA 签名验证 |
| 安全启动 | 机器人端 Secure Boot |
| 通信加密 | MQTT over TLS，端到端加密 |

## 5. 可靠性需求

### 5.1 数据一致性

| 场景 | 一致性要求 | 实现方式 |
|------|-----------|----------|
| 任务状态变更 | 强一致 | 数据库事务 + 分布式锁 |
| 告警状态 | 强一致 | 数据库事务 |
| 机器人实时位置 | 最终一致 | Redis 缓存，允许短暂延迟 |
| 传感器数据 | 最终一致 | 异步写入 InfluxDB |
| 用户权限变更 | 强一致 + 缓存失效 | 数据库事务 + Redis 缓存清除 |

### 5.2 消息可靠性

| 消息类型 | 可靠性保证 | 机制 |
|----------|-----------|------|
| 任务指令 | At-least-once | MQTT QoS 1 + 应用层确认 |
| 告警事件 | At-least-once | Kafka ACK + 消费者幂等 |
| 遥测数据 | At-most-once | MQTT QoS 0（允许少量丢失） |
| 操控指令 | At-most-once | 低延迟优先，丢失即过期 |
| 状态变更 | Exactly-once | Kafka 事务 + 数据库幂等写 |

### 5.3 容错设计

- **重试策略**：指数退避（1s → 2s → 4s → 8s），最大重试 5 次
- **熔断器**：错误率 > 50% 触发熔断，10s 后半开探测
- **超时控制**：HTTP 30s、gRPC 10s、MQTT 5s
- **限流保护**：令牌桶算法，每用户每 API 限制
- **优雅降级**：非核心功能故障不影响核心巡检链路

## 6. 可维护性需求

### 6.1 日志规范

```json
{
    "timestamp": "2026-03-04T12:00:00.000Z",
    "level": "INFO",
    "service": "task-svc",
    "trace_id": "abc123",
    "span_id": "def456",
    "message": "Task started",
    "fields": {
        "task_id": 123,
        "robot_id": 5,
        "site_id": 1
    }
}
```

| 日志级别 | 使用场景 |
|----------|----------|
| ERROR | 需要立即关注的错误 |
| WARN | 异常但可恢复的情况 |
| INFO | 关键业务流程节点 |
| DEBUG | 开发调试信息（生产默认关闭） |

### 6.2 监控指标

**RED 方法**（面向请求的服务）：
- **Rate**：请求速率 (QPS)
- **Errors**：错误率
- **Duration**：延迟分布

**USE 方法**（面向资源）：
- **Utilization**：利用率
- **Saturation**：饱和度
- **Errors**：错误数
