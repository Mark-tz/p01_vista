# 智能巡检调度系统 — 系统架构总览

> 项目代号：VISTA（Versatile Inspection System with Tactile Arm）
> 版本：v0.1.0-draft
> 日期：2026-03-04

---

## 1. 系统定位

VISTA 是面向工业场景的**四足机器人智能巡检调度平台**，提供从任务编排、自主巡检、实时监控、远程操控到数据分析的全链路闭环能力。系统支持多机器人协同、多楼层巡检、AI 视觉识别、传感器数据采集等核心业务场景。

### 1.1 核心价值

| 维度 | 说明 |
|------|------|
| 安全替代 | 替代人工进入高危、高温、有毒等恶劣环境 |
| 效率提升 | 7×24 小时不间断巡检，消除人工排班瓶颈 |
| 数据驱动 | 自动采集、识别、归档，构建设备健康数据资产 |
| 快速响应 | 异常秒级告警 + 远程操控介入，缩短响应链路 |

## 3. 服务职责矩阵

| 服务 | 职责 | 通信方式 | 关键依赖 |
|------|------|----------|----------|
| **API Gateway** | 统一入口、路由、认证、限流 | HTTP/WS → gRPC | Redis（会话/限流） |
| **用户服务 (IAM)** | 用户管理、角色权限、审计日志 | gRPC | PostgreSQL |
| **地图服务 (Map)** | 地图数据管理、点位/区域/围栏 CRUD | gRPC | PostgreSQL, MinIO |
| **任务服务 (Task)** | 任务编排、调度引擎、执行控制 | gRPC + Kafka | PostgreSQL, Redis |
| **设备服务 (Device)** | 机器人资产管理、状态同步、充电调度 | gRPC + MQTT | PostgreSQL, EMQX |
| **监控服务 (Monitor)** | 实时状态推送、轨迹记录 | WebSocket + MQTT | Redis, InfluxDB |
| **告警服务 (Alarm)** | 规则引擎、告警生命周期、通知推送 | gRPC + Kafka | PostgreSQL, Redis |
| **数据服务 (Data)** | 传感器数据存储、报表生成、数据导出 | gRPC | InfluxDB, PostgreSQL |
| **媒体服务 (Media)** | 视频流代理、图片管理、转码 | HTTP + RTSP/WebRTC | MinIO, FFmpeg |
| **远程操控 (Remote)** | 运动控制指令下发、云台控制、接管 | WebSocket + MQTT | EMQX |
| **AI 服务 (AI)** | 图像识别、仪表读数、异常检测 | gRPC | GPU, MinIO |
| **OTA 服务** | 固件/软件版本管理、升级下发 | gRPC + MQTT | MinIO, PostgreSQL |

## 5. 通信协议矩阵

| 场景 | 协议 | 说明 |
|------|------|------|
| 前端 ↔ 网关 | HTTPS + WSS | REST API + WebSocket 实时推送 |
| 服务间调用 | gRPC (Protobuf) | 高性能、强类型内部通信 |
| 异步事件 | Kafka | 任务事件、告警事件、审计日志 |
| 设备通信 | MQTT (v5) via EMQX | 机器人状态上报、指令下发 |
| 视频传输 | RTSP → WebRTC | 低延迟实时视频 |
| 文件传输 | HTTP (MinIO S3 API) | 地图文件、固件包、图片 |

## 6. 关键架构决策记录（ADR）

### ADR-001: 采用微服务架构而非单体

- **背景**：系统涵盖实时监控、任务调度、AI 推理等差异化负载
- **决策**：微服务拆分，按业务域独立部署
- **理由**：不同服务有截然不同的资源需求（AI 需 GPU、视频需高带宽、调度需高可用），微服务允许独立扩缩容
- **风险**：运维复杂度增加，需完善的 DevOps 体系

### ADR-002: MQTT 作为设备通信协议

- **背景**：机器人在工业场景网络条件可能不稳定
- **决策**：使用 MQTT v5 + EMQX 作为设备通信层
- **理由**：MQTT 低带宽开销、支持 QoS 等级、断线重连、遗嘱消息，天然适合 IoT 场景
- **备选**：gRPC 直连（被否决：不适合弱网环境）

### ADR-003: 时序数据库存储传感器数据

- **背景**：传感器数据为高频时间序列，查询模式以时间范围为主
- **决策**：采用 InfluxDB 独立存储传感器数据
- **理由**：关系型数据库在高频写入 + 时间范围查询场景下性能不足
- **备选**：TimescaleDB（PostgreSQL 扩展，若希望减少运维组件可考虑）

### ADR-004: WebRTC 实现低延迟视频

- **背景**：远程操控要求视频延迟 < 500ms
- **决策**：RTSP → WebRTC 转码推流
- **理由**：WebRTC 端到端延迟可控制在 200-500ms，浏览器原生支持
- **备选**：RTMP + FLV.js（延迟 1-3s，不满足操控要求）

### ADR-005: 事件驱动的告警架构

- **背景**：告警来源多样（设备异常、AI识别、环境超限），需解耦处理
- **决策**：所有告警事件通过 Kafka 统一收集，由告警服务消费处理
- **理由**：解耦告警生产者与消费者，支持告警合并、去重、升级等复杂策略

## 8. 安全架构概要

```
                    ┌─────────────────┐
                    │    WAF / DDoS   │
                    └────────┬────────┘
                             │
                    ┌────────▼────────┐
                    │   TLS 终结       │
                    │   API Gateway   │
                    │   JWT 验证       │
                    │   RBAC 鉴权      │
                    └────────┬────────┘
                             │ mTLS
              ┌──────────────┼──────────────┐
              │              │              │
        ┌─────▼─────┐ ┌─────▼─────┐ ┌─────▼─────┐
        │  服务 A   │ │  服务 B   │ │  服务 C   │
        │(服务网格)  │ │(服务网格)  │ │(服务网格)  │
        └───────────┘ └───────────┘ └───────────┘
              │              │              │
        ┌─────▼──────────────▼──────────────▼─────┐
        │         数据层加密（TDE / 字段加密）        │
        └─────────────────────────────────────────┘
```

- **传输安全**：外部 TLS 1.3，内部 mTLS（服务网格）
- **认证**：JWT + Refresh Token，设备端 X.509 证书
- **鉴权**：RBAC（管理员 / 运维 / 访客），资源级权限控制
- **审计**：全链路操作日志，不可篡改存储
- **数据安全**：敏感字段加密，数据库 TDE，定期备份
