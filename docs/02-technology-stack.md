## 1. 选型原则

| 原则 | 说明 |
|------|------|
| **成熟稳定** | 优先选择经过大规模生产验证的技术 |
| **生态完善** | 社区活跃、文档齐全、人才储备充足 |
| **性能适配** | 匹配场景需求（实时性、高并发、大数据量） |
| **运维友好** | 支持容器化部署，可观测性好 |
| **许可合规** | 优先 Apache 2.0 / MIT 等宽松许可 |

### 前端架构补充说明

- **微前端**：暂不引入。如后续团队规模扩大到多团队并行开发，可考虑 qiankun / Module Federation

- **离线能力**：巡检地图等关键资源支持 Service Worker 缓存，弱网下可降级展示

## 4. 数据存储

| 组件 | 选型 | 用途 | 选型理由 |
|------|------|------|----------|
| 关系型数据库 | **PostgreSQL** 16 | 用户/任务/设备/告警等业务数据 | ACID 完整、JSON 支持好、扩展能力强 |
| 缓存 | **Redis** 7.x (Cluster) | 会话、限流、实时状态、分布式锁 | 亚毫秒延迟，数据结构丰富 |
| 时序数据库 | **InfluxDB** 3.x | 传感器数据、设备指标 | 高频写入优化、内置降采样、类 SQL 查询 |
| 对象存储 | **MinIO** | 地图文件、图片、视频、固件包 | S3 兼容、自托管、高性能 |
| 搜索引擎 | **Elasticsearch** 8.x | 日志检索、告警全文搜索 | 视数据规模，MVP 阶段可延后引入 |

### 存储选型矩阵

```
数据类型              存储引擎          理由
─────────────────────────────────────────────────
业务结构化数据         PostgreSQL       事务一致性、关联查询
实时状态/缓存          Redis            亚毫秒读写
传感器时序数据         InfluxDB         高频写入、时间范围查询
图片/视频/文件         MinIO            大对象、S3 兼容
操作审计日志           PostgreSQL       结构化、可审计
全文检索(可选)         Elasticsearch    非结构化检索
```

## 5. 消息与通信

| 组件 | 选型 | 用途 | 选型理由 |
|------|------|------|----------|
| 消息队列 | **Kafka** 3.x (KRaft) | 服务间异步事件（告警、任务状态、审计） | 高吞吐、持久化、精确一次语义 |
| MQTT Broker | **EMQX** 5.x | 机器人设备通信 | 百万级连接、规则引擎、桥接 Kafka |
| WebSocket | **Socket.IO** (Go 实现) | 前端实时推送 | 自动重连、房间/命名空间 |

### MQTT Topic 设计规范

```
vista/                              # 顶层命名空间
├── robot/{robot_id}/               # 按机器人分组
│   ├── status                      # 状态上报 (QoS 1)
│   ├── telemetry                   # 遥测数据 (QoS 0)
│   ├── event                       # 事件上报 (QoS 1)
│   ├── cmd/{cmd_type}              # 指令下发 (QoS 1)
│   └── video/control               # 视频控制 (QoS 1)
├── task/{task_id}/                  # 按任务分组
│   ├── progress                    # 进度上报
│   └── result                      # 结果上报
└── system/                         # 系统级
    ├── broadcast                   # 广播消息
    └── ota/{robot_id}              # OTA 升级
```

## 6. 视频流方案

### 架构方案

```
机器人摄像头 ──RTSP──► 媒体服务器 ──┬── WebRTC ──► 实时预览（低延迟）
                                   ├── HLS ────► 录像回放（兼容性好）
                                   └── 抓拍 ───► MinIO（图片存储）
```

| 场景 | 协议 | 延迟 | 说明 |
|------|------|------|------|
| 实时预览 | WebRTC | 200-500ms | 远程操控时的主视角 |
| 多路监控 | WebRTC / HLS | 500ms-3s | 多画面巡视可适当放宽延迟 |
| 录像回放 | HLS / HTTP-FLV | — | 分片存储，按需拉流 |
| 视频录制 | RTSP → FFmpeg → MinIO | — | 服务端录制，MP4 分段存储 |

### 媒体服务器选型

| 候选 | 优势 | 劣势 |
|------|------|------|
| **ZLMediaKit** | 国产开源，RTSP/WebRTC/HLS 全协议支持，性能优异 | 文档偏少 |
| **MediaMTX** | Go 实现，轻量，RTSP→WebRTC 转换简单 | 功能相对简单 |
| **SRS** | 功能全面，社区活跃 | 偏向直播场景 |

**推荐**：ZLMediaKit 作为主方案，MediaMTX 作为轻量备选。

## 9. 技术风险与缓解

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| WebRTC 弱网下卡顿 | 远程操控体验下降 | 自适应码率 + SVC 分层编码，降级到低分辨率 |
| MQTT 高并发消息积压 | 设备指令延迟 | EMQX 集群扩展 + 消息优先级 + 过期丢弃策略 |
| AI 推理延迟过高 | 告警不及时 | 边缘推理（机器人端）+ 云端复核双模式 |
| 多楼层地图数据量大 | 前端渲染性能问题 | LOD 分层加载 + Web Worker 离屏渲染 |
| 微服务运维复杂 | 故障排查困难 | 完善可观测性（指标+日志+追踪三支柱） |
| 视频存储空间增长快 | 存储成本高 | 分级存储策略（热/温/冷）+ 自动清理 |
